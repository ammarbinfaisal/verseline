# Recitation Clip Workflow

This project can be reorganized around one specific job:

1. A portrait reel.
2. One background image or looped background video.
3. One recitation audio track.
4. One small meta block.
5. One timed translation block.

The current VOD/chat/chunk structure is not the right mental model for this. For Quran recitation clips, the timeline should be driven by the audio and the subtitle segments, not by source-video cuts.

This document assumes the tool is being narrowed for one workflow only:

- portrait recitation reels
- one translation shown on screen
- exact translation wording preserved
- generated draft first, then manual approval, then final render

## Core Principle

The main unit should be a `segment`, and a segment should mean:

- `from this timestamp`
- `to this timestamp`
- `show this translation text`

Verse references are useful, but they should be secondary metadata.

This matters because your real workflow is often:

- one verse split into 2-4 subtitle segments
- one subtitle segment covering only part of a verse
- one subtitle segment aligned to a pause in the recitation rather than a verse boundary
- occasional bracketed explanatory text that should be visually de-emphasized

So the correct model is:

- audio timing is the source of truth
- displayed translation text is the source of truth for the segment
- verse references are attached metadata used for lookup, traceability, and later editing
- transcript output is only a draft aid, not final truth

Do not force `1 segment = 1 verse`.
Do not force the visible English text to be regenerated by the model.

## Recommended Scope

If this repo is only for your recitation workflow, it is reasonable to remove or ignore:

- Twitch chat support
- VOD/chunk tooling
- cut-preview features tied to stream editing
- stack-language features that only exist for the old workflow

The streamlined tool can become:

1. `validate`
2. `render`
3. `preview` or `render-segment`

That is enough.

## Recommended File Model

Use three layers.

### 1. Project Defaults

This is per clip, or shared across many clips.

It should contain:

- output size: `1080x1920`
- safe margins
- default font
- default colors
- background asset
- audio asset
- meta info
- translation source file
- segment draft file
- approved segment file

Example:

```json
{
  "output": "out/al-ikhlas-reel.mp4",
  "audio": "audio/al-ikhlas.mp3",
  "background": {
    "type": "image",
    "path": "assets/backgrounds/blue-sky.jpg"
  },
  "font": "Aptos",
  "meta": {
    "surah": "Al-Ikhlas",
    "verse_range": "1-4",
    "translation": "Sahih International"
  },
  "layout": {
    "safe_margin_x": 108,
    "translation_margin_bottom": 360,
    "meta_margin_bottom": 540
  },
  "translation_source": {
    "path": "data/translations.sahih.jsonl",
    "format": "jsonl"
  },
  "segments_draft_file": "clips/al-ikhlas/segments.draft.json",
  "segments_file": "clips/al-ikhlas/segments.json"
}
```

## 2. Translation Source

Use this only as a lookup table for full-verse reuse.

Recommended storage:

- `JSONL` for large verse datasets
- `JSON` for smaller hand-maintained datasets

SQLite is only worth it if you need cross-project querying, multiple translations at scale, or complex tooling. For your current workflow, `JSONL` is simpler and easier to inspect and edit.

Recommended record shape:

```json
{"surah":112,"ayah":1,"text":"Say, He is Allah, [who is] One,"}
{"surah":112,"ayah":2,"text":"Allah, the Eternal Refuge."}
```

## 3. Timeline Segments

This is the most important file in the whole workflow.

Each segment is an audio-timed display unit.

Example:

```json
[
  {
    "start": "00:00:00.000",
    "end": "00:00:03.900",
    "verse": "112:1"
  },
  {
    "start": "00:00:03.900",
    "end": "00:00:06.800",
    "text": "Say, He is Allah, [who is] One,",
    "refs": ["112:1"]
  },
  {
    "start": "00:00:06.800",
    "end": "00:00:10.200",
    "text": "Allah, the Eternal Refuge.",
    "refs": ["112:2"],
    "size": "large"
  }
]
```

The important distinction is:

- `verse` means: auto-fill from the translation source because this segment is exactly one full verse
- `text` means: manual text because this segment is only part of a verse, a merged phrase, or tuned to the recitation pacing
- `refs` means: optional metadata for traceability even when text is manual

That split is the right solution for your problem.

## Segment Rules

Each segment should support at least:

- `start`
- `end`
- `text` or `verse`
- `size`
- `refs`

Recommended meaning:

- `start` and `end`: always required
- `text`: manual display text
- `verse`: auto lookup when the segment maps cleanly to one verse
- `refs`: one or more verse references connected to this segment
- `size`: `default` or `large`

Optional future fields:

- `notes`
- `id`
- `font_size_delta`
- `placement`
- `line_breaks`
- `status`
- `confidence`
- `source`

## What To Do When A Segment Does Not Match A Verse

This is the normal case, not an edge case.

When the segment does not match a full verse, do this:

1. Enter the final display text manually.
2. Add `refs` for the underlying verse or verses.
3. Let the timing follow the audio, not the verse boundary.

Example:

```json
{
  "start": "00:00:12.000",
  "end": "00:00:15.200",
  "text": "And do not mix the truth with falsehood",
  "refs": ["2:42"]
}
```

If a segment combines pieces from multiple verses:

```json
{
  "start": "00:00:15.200",
  "end": "00:00:19.600",
  "text": "while you know [the truth].",
  "refs": ["2:42", "2:43"]
}
```

This is better than inventing a complicated verse-slicing syntax too early.

## Bracketed Text

Bracketed text should stay inside the segment text itself.

Example:

```json
{
  "text": "Say, He is Allah, [who is] One,"
}
```

The renderer should detect bracketed spans and automatically style them with:

- a softer color
- optionally slightly lower opacity
- no other markup required from you

That keeps the authoring format simple.

## Whisper and Alignment

Yes, you can use Whisper, but it should be used as a drafting and alignment aid, not as the authority for the final wording.

The practical role of Whisper is:

- detect rough phrase boundaries
- detect pauses and breath points
- produce a draft transcript or transliterated cue stream
- help place timestamps for likely segment breaks

The practical role of the translation source is:

- provide the exact English text you are allowed to show
- preserve the chosen translation wording exactly

So the system should work like this:

1. Run Whisper on the recitation audio.
2. Extract time-stamped phrase candidates.
3. Map those phrase candidates to exact substrings of the selected translation.
4. Generate a draft segment file.
5. Review and approve the draft.
6. Render only from the approved segment file.

That means Whisper is not writing the final subtitles.

It is only helping to propose segment boundaries.

## Exact-Text Policy

This should be a hard rule:

- the renderer may only show text that already exists in the chosen translation source

So if the qari stops at a point that corresponds to `those`, then the visible segment should use the exact substring `those` from the translation text, not a paraphrase, not a synonym, and not an LLM rewrite.

This is important because your real requirement is not just semantic alignment.

It is textual fidelity to a chosen translation.

That implies two allowed cases only:

### Case 1. Exact Full Verse

The segment maps to a whole verse.

Use:

```json
{
  "start": "00:00:00.000",
  "end": "00:00:03.900",
  "verse": "2:5"
}
```

### Case 2. Exact Substring of One or More Verses

The segment maps to only part of the translation.

Use:

```json
{
  "start": "00:00:03.900",
  "end": "00:00:06.800",
  "text": "those are",
  "refs": ["2:5"],
  "source": {
    "type": "substring",
    "from": "2:5"
  }
}
```

This should be traceable back to the verse text, even when the segment is partial.

## Recommended Draft and Approval Flow

There should be two segment files:

- `segments.draft.json`
- `segments.json`

The first is generated.

The second is approved and edited by you.

Only `segments.json` should ever be rendered.

Recommended statuses inside the draft file:

- `draft`
- `approved`
- `needs_fix`

Example:

```json
{
  "start": "00:00:12.000",
  "end": "00:00:15.200",
  "text": "those are upon [right] guidance",
  "refs": ["2:5"],
  "size": "default",
  "status": "needs_fix",
  "confidence": 0.72,
  "notes": "boundary after Arabic pause sounds slightly early"
}
```

The right review loop is:

1. Generate draft.
2. Open the draft in a simple editor or review UI.
3. Fix segment boundaries, size, and line breaks.
4. Approve segments.
5. Render.

That will be much more reliable than trying to render directly from model output.

## Layout Recommendation

Keep both text blocks away from platform UI zones.

Recommended defaults for `1080x1920`:

- left/right safe margin: `96-120px`
- translation block: lower-middle, not flush to bottom
- meta block: above translation

Practical default:

- translation bottom margin: about `320-420px`
- meta bottom margin: about `500-620px`

This is usually safer for Instagram Reels and YouTube Shorts than placing text too low.

## Placement Strategy

The text position should not be literally random on every render or every frame.

It should be selected from a constrained set of safe candidate positions, then scored for contrast and readability.

That gives you variety without chaos.

Recommended approach:

1. Define 4-8 candidate translation anchors inside the safe area.
2. Define 2-4 candidate meta anchors above the translation area.
3. For each candidate, estimate background contrast under the text box.
4. Reject placements that overlap with platform-danger zones or each other.
5. Choose the highest-scoring candidate, or choose randomly among the top few.
6. Keep the chosen placement stable for the whole clip, or at most for a whole segment group.

Do not reposition text continuously.

That would look unstable and distracting.

Recommended fields:

```json
{
  "placement": {
    "mode": "auto",
    "seed": 12,
    "translation_candidates": ["lower_center", "lower_left", "mid_center"],
    "meta_candidates": ["above_translation", "upper_left_safe"]
  }
}
```

If needed, a segment can still override placement manually:

```json
{
  "start": "00:00:06.800",
  "end": "00:00:10.200",
  "text": "Allah, the Eternal Refuge.",
  "placement": "lower_center"
}
```

## Font and Style Recommendation

Keep one default font per workflow.

Only allow per-segment variation for:

- `default`
- `large`

Do not turn every segment into a styling exercise.

Recommended approach:

- one default font family
- one normal translation size
- one larger emphasis size
- one tiny meta size
- one primary text color
- one auxiliary bracket color
- one outline/shadow style

This is enough for consistency and speed.

## Two-Size Rule

You only need two discrete translation sizes:

- `default`
- `large`

Everything else should be a rare override.

Suggested usage:

- `default`: standard segment
- `large`: short segment, emphatic segment, or visually sparse moment

Meta text should always stay at one tiny fixed size.

So the style system should be:

- one default translation size
- one large translation size
- one fixed meta size
- optional one-off `font_size_delta` for correction

This keeps editing fast.

## Suggested Authoring Workflow

Use this order every time:

1. Choose audio.
2. Choose background.
3. Set clip-level meta.
4. Run Whisper to get a draft timing transcript.
5. Map the draft to exact translation substrings.
6. Generate `segments.draft.json`.
7. Review and approve into `segments.json`.
8. Render.

The critical points are step 4 and step 7:

- create the timeline from recitation pacing first
- always manually approve before render

That is the part that actually determines whether the reel feels right.

## How To Segment Very Long Recitation Runs

Sometimes a qari may continue for a long stretch without a clean short pause.

In that case, segmenting should not depend on pause detection alone.

Use these priorities in order:

1. Major recitation pause.
2. Natural completion of a translation phrase.
3. Readability length on screen.
4. Visual rhythm of the reel.

That means one long recited run may still need to be broken into 2-4 English segments even if the breath pattern alone would not force it.

Practical rule:

- prefer segments of one short clause or one medium clause
- avoid overly dense full-sentence blocks unless the timing clearly supports them
- when in doubt, split at punctuation or clause boundaries inside the exact translation text

This is where the LLM can help propose candidates, but the final approved text must still be an exact substring of the source translation.

## Mapping Whisper Output To Translation Text

This is the hardest part of the workflow.

The problem is:

- Whisper gives approximate Arabic or phonetic phrase timing
- the English translation is not word-for-word in order
- the qari may stop at a point that corresponds only loosely to an English clause boundary

So the tool should not ask the model to "translate the transcript".

It should ask the model to do constrained alignment.

The model's task is:

1. look at the time-stamped recitation draft
2. look at the exact translation text for the relevant verse range
3. propose segment boundaries
4. assign each segment an exact substring from the translation source
5. never invent or rewrite text

The model output should therefore be a segmentation proposal, not prose.

## Suggested Prompt Strategy For The LLM

Use a constrained prompt with explicit rules.

Recommended rules:

- you are not allowed to paraphrase
- you are not allowed to rewrite punctuation
- you may only return exact substrings from the supplied translation text
- every returned segment must be contiguous text from the source
- if confidence is low, mark it low instead of guessing confidently
- prefer boundaries that align with both recitation pauses and readable English clauses
- if a very long run must be split, choose the least awkward clause break

Suggested prompt skeleton:

```text
Task:
Align the recitation draft to the provided translation and propose subtitle segments.

Rules:
1. Do not paraphrase, summarize, or rewrite any translation text.
2. Every segment text must be an exact contiguous substring copied from the supplied translation source.
3. Keep segment boundaries aligned as closely as possible to recitation pauses, but prioritize readable English clause boundaries when needed.
4. Return low confidence when the alignment is uncertain.
5. Return JSON only.

Input:
- Recitation draft with timestamps
- Verse references in scope
- Exact translation text for those verses
- Maximum subtitle sizes allowed: default, large

Output JSON schema:
{
  "segments": [
    {
      "start": "HH:MM:SS.mmm",
      "end": "HH:MM:SS.mmm",
      "text": "exact substring only",
      "refs": ["2:5"],
      "size": "default",
      "confidence": 0.0,
      "reason": "short explanation"
    }
  ]
}
```

This is better than a broad prompt because it narrows the model to a verification and alignment role.

## Suggested Minimal CLI

If you simplify the app for this job, the CLI can be:

### `draft`

Runs:

- Whisper transcription
- alignment to translation
- draft segment generation

Outputs:

- `segments.draft.json`

### `render`

Renders the final reel from:

- clip config
- approved segment file
- translation source

### `validate`

Checks:

- timestamps are valid
- no empty segments
- no missing translation lookups
- all assets exist
- layout values are reasonable
- every visible string is either a full verse or an exact substring from the source translation

### `preview --segment N`

Renders only one segment or a small time window around it for quick review.

### `approve`

Copies or promotes approved draft segments into the final segment file.

That is a better fit than the current chunk/cut terminology.

## Future Helpers Worth Adding

These are useful later, but should not block the first clean version.

### 1. Verse Lookup Fill

If a segment has:

```json
{"verse":"112:1"}
```

the tool fills `text` automatically from the translation source.

### 2. Segment Preview

Render just one segment with a few seconds of padding.

This will save time while tuning text and timing.

### 3. Segment Import Template

Generate a starter segment file from rough timestamps so you only edit text and timing.

### 4. Multi-Translation Comparison

For difficult passages, quickly compare several translations before picking the final displayed wording.

### 5. Exact Substring Validation

Verify that every manual segment text is an exact substring of the referenced translation source.

### 6. Auto Placement Scoring

Choose text positions from safe candidates based on background contrast.

## What I Would Actually Build First

If we are rewriting this for your needs, I would build this in order:

1. A new clip config loader.
2. A translation lookup loader for `JSON` and `JSONL`.
3. A draft segment schema and approved segment schema.
4. Whisper-based draft transcript generation.
5. Constrained LLM alignment from transcript to exact translation substrings.
6. Approval flow for draft to final segments.
7. ASS subtitle generation with bracket styling.
8. ffmpeg render for portrait output with safe-area-aware placement.
9. `validate`.
10. `preview --segment`.

I would not start with SQLite.

I would not start with a complex markup language.

I would not make verse references the primary timing model.

I would not let the LLM freely rewrite translation text.

## Final Recommendation

The cleanest mental model for your workflow is:

- one clip config
- one audio track
- one background
- one generated draft timeline
- one approved timeline
- one optional verse lookup dataset

And the most important rule is:

Segments should map to how the recitation is heard, not to how the mushaf is indexed.

The second most important rule is:

Every visible English segment should come from the chosen translation exactly, even when the timing is generated from audio and model assistance.

That is the key to making this fast to author, reliable to review, and pleasant to watch.
